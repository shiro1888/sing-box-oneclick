# 节点搭建文档

这是一份通用节点搭建文档，不绑定某一台 VPS。以后新买 VPS、换 IP、换域名，都按这份模板改变量即可。

核心目标：

```text
一台自用 VPS
四条协议节点（Hysteria2 / AnyTLS / Vless / SS2022）+ 可选第 5 条 CF-Vless 大保底
一个 Clash/Mihomo 订阅
一个手动选择代理组
国内直连，国外代理
客户端显示订阅流量（有限流量可限流停机，无限流量只显示不停）
配置简单，可排查，可迁移
```

> 本版本相对早期版本做了若干健壮性优化：流量统计支持双向计费、运行参数统一到单一来源 `/etc/sing-box-node.env`、手动停机不会被定时任务拉起、补齐防火墙/时间同步/备份等步骤。改动原因见文末「之前改动抽象成通用规则」。

## 使用方式

如果是手机 Clash/Mihomo 导入节点，只需要订阅 URL（`SUB_PATH` 已自带开头 `/`，所以这里不再多写斜杠）：

```text
http://SUB_DOMAIN_1SUB_PATH
```

客户端里的订阅名称建议填机场 / 线路名字：

```text
AIRPORT_NAME
```

例如这条线路叫 `LINE-01`，客户端订阅名称就填 `LINE-01`。以后如果有多条线路，可以用 `US-01`、`JP-01`、`HK-01` 这种名字区分。

如果是新 VPS 部署，从“变量表”开始往下看。

如果是维护旧 VPS，先看“检查命令”和“修改前备份”。

如果只是想知道最终原则，看这一段：

```text
订阅路径用随机路径，不用 /sub.yaml
代理组用 select，不用 fallback
节点名只保留协议名
Subscription-Userinfo 只放在订阅路径
Nginx 首页和旧路径返回 404
443 留给 VLESS Reality，不做 HTTPS 订阅（默认）
运行参数集中放在 /etc/sing-box-node.env，脚本统一读取
流量统计默认按 tx（只算出站）计，匹配绝大多数商家的计费口径
手动停 sing-box 不会被定时任务拉起
Google Play 专用规则不写进通用模板
Cloudflare / Argo / WARP 不作为默认主线
如果要大保底，使用 CF-Vless-WS，并且只在 Cloudflare Tunnel 绑定后加入订阅
```

## 部署前须知

### 机器要求（最低）

```text
系统：推荐 Debian 11+ / Ubuntu 20.04+，必须有 systemd（Alpine/OpenRC 不支持）。
      本文命令按 Debian/Ubuntu 写（apt、cron、/etc/nginx/sites-enabled）。
      RHEL 系（Rocky/Alma/CentOS 8+）能跑但未逐条验证：需自行把 apt 换 dnf、
      装 cronie 代替 cron、nginx 配置路径改成 /etc/nginx/conf.d/。
架构：x86_64 或 arm64
配置：1 核 / 512MB 内存 / 5GB 硬盘足够（自用代理很轻，瓶颈是带宽不是配置）
权限：root（或能 sudo）
网络：能访问外网装包（apt/dnf + sing-box 官方源）
域名：可选，不强制（见下）
```

### 要开哪些端口（云安全组 + 本机 ufw 两层都要放行）

云安全组（服务商控制台：阿里云 / 腾讯云 / Oracle / AWS / Vultr 等）是**最容易漏的一层**，它默认拒绝入站、而且经常单独拦 UDP。下面这些是“按本手册默认端口”需要放行的：

| 端口 | 协议 | 用途 | 是否必开 |
|---|---|---|---|
| `22` | tcp | SSH | 必开（改过 SSH 端口就放你那个） |
| `80` | tcp | Nginx 订阅 | 必开 |
| `443` | tcp | VLESS Reality | 必开 |
| `4433` | **udp** | Hysteria2 | 必开（**最容易漏的就是这条 UDP**） |
| `4434` | tcp | AnyTLS | 必开 |
| `4435` | tcp + udp | Shadowsocks-2022 | 主线默认要开（TCP、UDP 都要放）；只有删掉 SS2022 才可不放 |
| 端口段（如 `20000-21000`） | udp | HY2 端口跳跃 | 仅启用端口跳跃才需要，要放**整段 UDP** |

> **端口段语法别照抄错**：上表的 `20000-21000`（连字符）是给**云安全组控制台**和 nftables 用的写法；**ufw 要用冒号** `ufw allow 20000:21000/udp`（ufw 不认连字符范围，照抄会报错）。

> **不要对外开**的端口：CF-Vless 的本地入站 `127.0.0.1:28080`、网页管理面板 `127.0.0.1:8088`——它们只监听本机，靠 SSH 隧道访问，**绝不要**在安全组 / ufw 放行。
>
> 自检口诀：本机 `ss -lntup` 只能证明“本地在监听”，证明不了“外部能连进来”。**“HY2 / SS 连不上、但 Vless 正常”99% 是云安全组没放对应 UDP。**

### 域名要不要

```text
不强制：裸 IP 也能用（订阅 URL 和节点 server 都可以直接写 IP）。
有域名的好处：订阅地址好记、能走 Cloudflare 加密订阅、纯 IPv6 机器更好用。
本手册示例多用域名 SUB_DOMAIN_1；纯裸 IP 部署时：
  - 订阅 server 的 server_name 直接填这台机的 IP；
  - DNS 那一步跳过；
  - 订阅 URL 用 http://<本机IP>SUB_PATH（SUB_PATH 同样整篇替换成真实路径）。
```

### 关于流量限制（不是每台机器都要限）

机器分两种，**按你这台的实际情况选**，不要照抄一个固定额度：

```text
有限流量（多数 VPS，按月给额度）：
   按「运行参数」把 LIMIT_GB 设成你的真实额度（建议留点缓冲，见该节换算说明），
   超额会自动停 sing-box，避免产生超额费 / 被停机。

无限流量（不限量套餐 / 大流量小鸡）：
   你不想被停机，两种做法二选一——
   ① 把 LIMIT_GB 设得很大（例如 999999），限流脚本就只「显示用量」、永不停机；（推荐）
   ② 完全不要流量显示：不仅跳过 traffic_limit.py，还要删掉 Nginx 段里生成的
      /etc/nginx/snippets/sub_headers.conf 和 server 块里的 include 那一行，再 reload nginx。
      若只跳过 traffic_limit.py 而保留 Nginx 的 include，客户端会一直显示 download=0
      （占位头不会被刷新），并不是「不显示流量」。
```

> 即便是无限流量，**留着限流脚本 + 设个大额度**仍有用：客户端能看到本月用了多少，方便发现异常（比如被人盗用订阅猛跑流量）。所以推荐做法①。

### 部署顺序与“完成度”（重要，别把可选件当成已部署）

这份文档分**主线**和**可选增强**两层，别混为一谈：

```text
主线（本文档纯手工就能搭完，搭完即可用）：
   四协议节点(HY2 / AnyTLS / Vless / SS2022) + Mihomo 订阅 + 国内直连分流

可选增强（没做完之前，不算“已部署”；确认跑通后再纳入常用配置）：
   CF-Vless 大保底（= Argo 隧道）、WARP 解锁、看板页、网页管理面板、BBRv3 换内核 等
```

推荐执行顺序（从稳到险，一步跑通再上下一步）：

```text
1. 先把四协议主线跑通（sing-box check 过、订阅能拉、客户端能连）
2. 再加流量显示/限流 + 订阅传输保护（随机路径，必要时 CF 加密订阅）
3. 网络加速第一步 sysctl（16MB UDP 缓冲，对 HY2 收益最大，免重启）
4. 最后再做高风险增强：BBRv3 按本文性能优先路线走 byJoey Max + 菜单 12；CF 大保底 / WARP 按需加
```

> 特别提醒 **CF-Vless 大保底（Argo 隧道）**：它要等到 `cloudflared` 装好、`127.0.0.1:28080` 本地入站加好、Tunnel 绑定、`curl` 验证看到 `101 Switching Protocols` **全部完成**，才算真的部署好、才把这个节点加进订阅。在这之前它就是个**死节点**，加进订阅只会让客户端多一条连不上的线路。

## 变量表

部署前先确定这些值。分成两类。

### A. 必填占位符（主线必填，IP 类按机器 IP 栈调整）

文档正文里凡是出现这些大写 token，都用真实值做整篇查找替换。其中 `SERVER_IP` / `SERVER_IPV6` / `SERVER_ADDRESS` 按 IPv4-only / IPv6-only / 双栈分情况填（见「IPv4 / IPv6 / 双栈」一节），不是每台都三个全填。

| 变量 | 说明 | 示例 |
|---|---|---|
| `AIRPORT_NAME` | 机场 / 线路在客户端里的订阅名称 | `LINE-01` |
| `SERVER_IP` | VPS 公网 IPv4（IPv4 / 双栈机器的 `IP-CIDR/32` 分流规则引用它；纯 IPv6 改用 `IP-CIDR6,SERVER_IPV6/128`） | `1.2.3.4` |
| `SERVER_ADDRESS` | 节点 `server:` 实际连接地址。**IPv4 默认填 `SERVER_IP` 的值**；纯 IPv6 填域名 `SUB_DOMAIN_1` | `1.2.3.4` |
| `SERVER_IPV6` | VPS 公网 IPv6，仅 IPv6 / 双栈用到（分流规则的 `IP-CIDR6/128` 引用它） | `2001:db8::1` |
| `SUB_DOMAIN_1` | 主订阅域名 | `node.example.com` |
| `SUB_DOMAIN_2` | 备用订阅域名，没有就删掉相关规则 | `node2.example.com` |
| `SUB_PATH` | 随机订阅路径 | `/sub-a8f3c91d7e4b4c6f.yaml` |
| `HY2_PASSWORD` | Hysteria2 密码 | `openssl rand -hex 16` |
| `ANYTLS_PASSWORD` | AnyTLS 密码 | `openssl rand -hex 16`（与上面不同的一条） |
| `VLESS_UUID` | VLESS UUID | `sing-box generate uuid` |
| `REALITY_PRIVATE_KEY` | Reality 私钥，服务端用 | `sing-box generate reality-keypair` |
| `REALITY_PUBLIC_KEY` | Reality 公钥，客户端用 | 同上命令的 PublicKey |
| `REALITY_SHORT_ID` | Reality short-id | `openssl rand -hex 8` |
| `SS_PASSWORD` | Shadowsocks-2022 密码（base64，**不是 hex**） | `openssl rand -base64 16`（128-gcm 用 16 字节；256/chacha 用 32） |

运行参数（流量/到期/网卡/计费方式）不在这里手填，统一写进 `/etc/sing-box-node.env`，见“运行参数：单一来源”一节。

### B. 推荐默认值（一般保留，要改需多处同步）

这些值模板里直接写成了推荐默认，不是查找替换的占位符。保留即可；**如果要改，下面每个位置都要同步改**，否则服务端和客户端会对不上。

| 变量 | 默认值 | 出现位置 |
|---|---|---|
| `HY2_PORT` | `4433/udp` | sing-box 入站、Clash 节点、防火墙、端口检查 |
| `ANYTLS_PORT` | `4434/tcp` | sing-box 入站、Clash 节点、防火墙、端口检查 |
| `VLESS_PORT` | `443/tcp` | sing-box 入站、Clash 节点、防火墙、端口检查 |
| `SS_PORT` | `4435/tcp+udp` | sing-box 入站、Clash 节点、防火墙、端口检查 |
| `SS_METHOD` | `2022-blake3-aes-128-gcm` | sing-box 入站 `method`、Clash 的 `cipher` |
| `REALITY_SNI` | `www.microsoft.com` | sing-box 的 `server_name` 和 `handshake.server`、Clash 的 `servername` |
| `TLS_SNI` | `www.bing.com` | 自签证书 CN、Clash 的 `sni` |

> 注意：`REALITY_SNI` 必须在「服务端 `handshake.server`」和「客户端 `servername`」两处完全一致，否则 Vless 节点会静默失败（连得上 TLS 但代理不通，很难排查）。改 SNI 时务必两处一起改。

## 运行参数：单一来源

流量额度、到期时间、统计网卡、计费方式统一放在一个文件，脚本和初始化都从这里读，避免“改了一处忘了另一处”。

```bash
# 先用 IPv4 默认路由探测出口网卡；纯 IPv6 机器 IPv4 探测会失败，再用 IPv6 兜底。
# 两者都失败才回退 eth0（网卡名为 ens3/enp1s0 等时，这一步能避免限流脚本首次报 interface not found）。
DETECTED_IF=$(ip route get 1.1.1.1 2>/dev/null | grep -oP 'dev \K\S+' | head -1)
[ -z "$DETECTED_IF" ] && DETECTED_IF=$(ip -6 route get 2606:4700:4700::1111 2>/dev/null | grep -oP 'dev \K\S+' | head -1)

cat >/etc/sing-box-node.env <<EOF
# 每月显示/限流额度。变量名 LIMIT_GB 与一键脚本保持一致，但单位实际是 GiB（脚本按 1024^3 字节算，不是十进制 GB）。
# 商家如果按十进制 GB 标流量，这里要换算：例如商家 200GB ≈ 186 GiB，填 186 才不会晚停约 7.4%。
# 无限流量的机器：把它设得很大（如 999999），脚本就只显示用量、永不停机（见「部署前须知 - 流量限制」）。
LIMIT_GB=200
# VPS / 订阅到期时间（带时区偏移，格式必须是 %Y-%m-%d %H:%M:%S %z）
# 注意：EXPIRE_AT 只用于客户端显示到期，限流脚本不会在到期时停服务（只有 LIMIT_GB 超额才真停）。
# 要到期自动停机，自己加一条 cron/at，或在脚本里加 now>expire 时 systemctl stop sing-box。
EXPIRE_AT="2026-12-31 23:59:59 +0800"
# 统计流量的网卡（自动探测，探测失败回退 eth0，可手动改）
INTERFACE=${DETECTED_IF:-eth0}
# 计费方式：tx=只算出站(默认,匹配绝大多数商家) / max=取较大方向 / rx+tx=双向相加(仅商家真按进+出计费才用)
COUNT_MODE=tx
EOF

chmod 600 /etc/sing-box-node.env
cat /etc/sing-box-node.env
```

`COUNT_MODE` 怎么选（默认 `tx`，对绝大多数机器是对的）：

```text
tx     只算出站（默认）。Vultr / DigitalOcean / AWS / GCP / Hetzner 等绝大多数商家
       都只按出站(outbound)计费——这个值正好等于你的真实用量，最贴合账单。
max    取进/出较大的一方。代理 rx≈tx，所以 max≈tx，比 tx 多一丁点安全余量。
rx+tx  把进+出相加。只有极少数商家真的按"进+出双向"计费，才用它。
```

> **为什么默认不是 rx+tx（重要纠正）**：代理是转发，每个字节过网卡两次——客户端下载 1GB，数据先从上游进 VPS（`rx`≈1GB）、再转发给客户端出 VPS（`tx`≈1GB），所以网卡上 `rx+tx`≈**2GB**。在"只按出站计费"的机器上（**绝大多数**），用 `rx+tx` 会把真实用量按 **2 倍**算，导致 `LIMIT_GB` 在真实用量约**一半**时就误停机、客户端流量也显示成 2 倍。所以默认用 `tx` 最贴合真实计费；**只有确认商家是按 `rx+tx` 双向计费，才改成 `rx+tx`**（这种商家其实很少）。

如果是 Google Cloud 这类主要看 Internet 出站免费额度的机器，可以用：

```text
COUNT_MODE=tx
LIMIT_GB=180
```

这样按出站流量限速，并给 200GiB 免费额度留约 20GiB 缓冲。不要卡到 200GiB 整数，因为云厂商账单和 `vnstat` 的统计口径、延迟、GiB/GB 显示可能不完全一致。

网卡名手动核对：

```bash
ip -br link
```

## 四个协议

客户端节点名只保留协议名：

```text
Hysteria2
AnyTLS
Vless
SS2022
```

实际协议组合：

```text
Hysteria2 = UDP/QUIC + TLS + h3 + SNI www.bing.com
AnyTLS    = TCP + TLS + SNI www.bing.com
Vless     = VLESS + TCP + Vision + uTLS chrome + REALITY + www.microsoft.com
SS2022    = Shadowsocks-2022（2022-blake3-aes-128-gcm，AEAD），TCP + UDP
```

使用建议：

```text
1. 优先试 Hysteria2，适合 UDP 正常的网络
2. HY2 不稳时切 AnyTLS，TCP 兼容性更好
3. 排查或兜底时切 Vless，Reality + Vision 稳定性好
4. 前面都不行再试 SS2022，协议简单、客户端兼容性最好，最不容易出兼容性问题
```

`Vless` 只是显示名。实际仍然是 `VLESS Reality Vision`，不是把 Reality 或 Vision 去掉了。

> 版本下限：AnyTLS 需要 **sing-box ≥ 1.12.0** 和 **mihomo ≥ 1.19.3**；SS2022 几乎所有现代客户端都支持。装完后用 `sing-box version` 确认，旧版没有 anytls 入站会直接 `sing-box check` 报错（这时可去掉 anytls 入站，保留 HY2 + Vless + SS2022 三条）。
>
> SS2022 的密码是 **base64**（`openssl rand -base64 16`），分享/填写时不要当成 hex；密码字节数要和加密方法匹配（128-gcm=16 字节，256-gcm / chacha20=32 字节），不匹配 `sing-box check` 会报错。

## 大保底协议

如果担心 VPS 直连 IP 被墙，推荐额外准备一条 Cloudflare 保底通道：

```text
CF-Vless = VLESS + WebSocket + Cloudflare Tunnel
```

> **CF-Vless 节点 和「Argo 隧道」节点是同一个东西，不是两个节点。**
> 「Argo Tunnel」是 Cloudflare 这条隧道的**旧名/别名**，现在官方叫「Cloudflare Tunnel」，底层都是 `cloudflared` 这个守护进程。本节用 `cloudflared` 建的隧道**就是** Argo 隧道，所以这一条 CF-Vless 节点 = 走 Argo/Cloudflare 隧道的 VLESS-WS 节点。**一个节点、两个叫法，不要再单独加一个「Argo 节点」，加了就是重复。**
>
> 另外，Cloudflare 隧道有两种，别混：
> ```text
> 命名隧道（Named Tunnel）：绑你自己的域名 CF_HOSTNAME，地址固定不变。
>                          本文档用的就是它，适合做订阅里的稳定保底节点。
> 临时隧道（Quick Tunnel / trycloudflare）：cloudflared tunnel --url ...，
>                          Cloudflare 随机给个 xxx.trycloudflare.com，免域名免账号，
>                          但每次重启域名都变、速度更差、随时可能失效。
>                          只适合临时测试，绝不能放进订阅当固定节点（URL 一变就成死节点）。
> ```
> 有些教程说的「Argo 节点」特指这个临时隧道——做保底不能用它，所以本文档坚持用**命名隧道**。

定位：

```text
平时不用
直连 IP 出问题时临时切换
速度不追求最高，优先保证能连
```

为什么选它：

```text
客户端连接 Cloudflare 域名，不直接连接 VPS IP
VPS 通过 cloudflared 主动连接 Cloudflare
即使 VPS IP 直连不可达，也有机会通过 Cloudflare 域名连接
和当前 VLESS / sing-box 体系一致，后续维护更顺
```

不建议把 Cloudflare 保底当主力：

```text
Cloudflare 链路更长
速度不一定比直连好
配置依赖 Cloudflare Tunnel 和域名路由
排查点比直连多
```

特殊 VPS 注意：

```text
如果 VPS 商家会对 Cloudflare / Tunnel / CDN 相关流量额外计费，或者明确不适合走 Cloudflare，
这一台就不要启用 CF-Vless。
做法是：订阅里不写 CF-Vless，sing-box 不保留 cf-vless-ws-in，cloudflared 停止并禁用。
其它普通 VPS 仍然可以按需使用 CF-Vless 作为大保底。
```

推荐最终节点排序：

```text
Hysteria2
AnyTLS
Vless
SS2022
CF-Vless
```

注意：只有 `CF_HOSTNAME` 已经绑定到 Cloudflare Tunnel 后，才把 `CF-Vless` 写进客户端订阅。否则客户端里会出现一个无法使用的死节点。

```bash
  ufw allow 22/tcp
```

```bash

```

```bash
sing-box generate uuid               # -> VLESS_UUID
sing-box generate reality-keypair    # PrivateKey -> REALITY_PRIVATE_KEY, PublicKey -> REALITY_PUBLIC_KEY
openssl rand -hex 16                 # -> HY2_PASSWORD
openssl rand -hex 16                 # -> ANYTLS_PASSWORD（再跑一次，要和上面不同）
openssl rand -hex 8                  # -> REALITY_SHORT_ID
openssl rand -base64 16              # -> SS_PASSWORD（base64！128-gcm 用 16 字节；若用 256-gcm/chacha 改成 -base64 32）
```

生成 HY2 / AnyTLS 自签证书（CN 用 TLS_SNI）：

```bash
mkdir -p /etc/sing-box
openssl req -x509 -nodes -newkey rsa:2048 \
  -keyout /etc/sing-box/server.key \
  -out /etc/sing-box/server.crt \
  -days 3650 \
  -subj "/CN=www.bing.com"

chmod 600 /etc/sing-box/server.key
chmod 644 /etc/sing-box/server.crt
```

> HY2 / AnyTLS 客户端模板里是 `skip-cert-verify: true`，所以自签证书 CN 不会被严格校验；这里把 CN 写成 `TLS_SNI` 只是为了配置一致、方便排查。真正必须两端完全一致的是 Reality 的 `REALITY_SNI`。

## 订阅传输安全

默认订阅走 `http://`（80 端口），订阅 YAML 里包含 UUID、密码、Reality 公钥等节点参数。明文 HTTP 下，能被动抓到这一次订阅请求的人就能拿到全套节点参数。随机路径是“藏”，不是“加密”。

自用、低频拉取、路径足够随机时，实际风险中等。按需选择加固：

```text
最低成本：保持随机 SUB_PATH，不外传，必要时换路径（只挡住别人继续拉订阅）
更稳妥：如果已经部署了下面的 Cloudflare Tunnel，把订阅也挂到 Cloudflare 走 HTTPS，订阅请求全程加密
```

> 注意：换 `SUB_PATH` **不等于换凭证**。订阅 YAML 里的 `HY2_PASSWORD`、`ANYTLS_PASSWORD`、`SS_PASSWORD`、`VLESS_UUID`、Reality 公钥一旦被抓走，旧路径失效后这些值依然有效、节点照样能被别人连。
> 所以一旦怀疑某次订阅被截获，要做的是**重置节点凭证**（重新生成上述密码/UUID，更新 sing-box 服务端与订阅 YAML 两边，`systemctl restart sing-box`），换路径只是顺手一起做。

> 重要：不能直接用 `CF_HOSTNAME` 拉订阅。下面「CF-Vless-WS 保底部署」里把 `CF_HOSTNAME` 整个回源到了 `http://127.0.0.1:28080`，那是 VLESS WS 入站，**不是** Nginx，订阅请求打过去拿不到 YAML。要走 Cloudflare 加密订阅，必须让订阅命中本机的 Nginx（80 端口），二选一：

```text
方式 A（推荐，最不容易错）：另起一个订阅专用域名 CF_SUB_HOSTNAME，
        在 Tunnel 里单独把它回源到 http://127.0.0.1:80，
        订阅地址用 https://CF_SUB_HOSTNAMESUB_PATH

方式 B（共用一个域名，按 path 分流）：仍用 CF_HOSTNAME，但在 Tunnel ingress 里
        按 path 拆：CF_WS_PATH 回源到 28080（WS 入站），其余（含 SUB_PATH）回源到 80（Nginx），
        订阅地址用 https://CF_HOSTNAMESUB_PATH
```

两种方式的 Tunnel 配置见下面「cloudflared」一节，已分别给出 ingress 模板。

无论 A 还是 B，cloudflared 回源到本机 80 时带的 Host（A 是 `CF_SUB_HOSTNAME`，B 是 `CF_HOSTNAME`）必须出现在 Nginx 订阅 server 的 `server_name` 里，否则会被默认 server 404。部署时把这个 Host 加进上一节 `server_name SUB_DOMAIN_1 SUB_DOMAIN_2 ...` 那行。

不建议为了订阅在 VPS 上单独开 443 HTTPS——443 留给 VLESS Reality（见“DNS 和客户端”一节说明）。

## Nginx 订阅服务

订阅头片段由脚本生成，初始值从 `/etc/sing-box-node.env` 读，不再硬编码：

```bash
mkdir -p /etc/nginx/snippets

# 用 env 里的单一来源生成初始头（download=0 占位，随后由 traffic_limit.py 刷新）
python3 - <<'PY'
import datetime
env = {}
for line in open("/etc/sing-box-node.env"):
    line = line.strip()
    if line and not line.startswith("#") and "=" in line:
        k, v = line.split("=", 1)
        env[k.strip()] = v.strip().strip('"').strip("'")
limit = int(float(env.get("LIMIT_GB", "200")) * 1024 ** 3)
# EXPIRE_AT 缺失或为空时回退 expire=0，和 traffic_limit.py 保持一致
expire_raw = env.get("EXPIRE_AT")
exp = int(datetime.datetime.strptime(expire_raw, "%Y-%m-%d %H:%M:%S %z").timestamp()) if expire_raw else 0
with open("/etc/nginx/snippets/sub_headers.conf", "w") as f:
    f.write(f'add_header Subscription-Userinfo "upload=0; download=0; total={limit}; expire={exp}" always;\n')
PY

chown root:root /etc/nginx/snippets/sub_headers.conf
chmod 644 /etc/nginx/snippets/sub_headers.conf
```

不要把流量头放到：

```text
/etc/nginx/conf.d/sub_headers.conf
```

原因是 Ubuntu Nginx 默认会全局加载 `conf.d/*.conf`，那样首页和 404 也会带 `Subscription-Userinfo`。

配置 Nginx：

拆成两个 server：默认 server 吃掉所有「Host 不对」或「直接用 IP 访问」的请求并一律 404；订阅 server 只绑定订阅域名，且只在精确的 `SUB_PATH` 上输出内容。这样即使有人猜中路径，用错 Host 或直接打 IP 也拿不到订阅。

```bash
cat >/etc/nginx/sites-enabled/default <<'EOF'
# 默认 server：兜底所有未匹配 Host（含直接用 IP 访问），一律 404，不暴露任何内容
server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _;
    return 404;
}

# 订阅 server：只对订阅域名生效，只在精确订阅路径上输出
server {
    listen 80;
    listen [::]:80;
    server_name SUB_DOMAIN_1 SUB_DOMAIN_2;

    root /var/www/html;

    location = SUB_PATH {
        include /etc/nginx/snippets/sub_headers.conf;
        default_type application/octet-stream;
        try_files $uri =404;
    }

    location / {
        return 404;
    }
}
EOF
```

注意：

- 上面 `location = SUB_PATH` 要替换成真实随机路径，例如 `location = /sub-a8f3c91d7e4b4c6f.yaml {`。这一步最容易漏，漏了订阅会 404 而 `nginx -t` 仍然通过。检查段有一条专门验证它有没有替换干净。
- `server_name SUB_DOMAIN_1 SUB_DOMAIN_2` 也要替换成真实域名。如果**没有**第二个订阅域名，删掉 `SUB_DOMAIN_2`，只留 `server_name SUB_DOMAIN_1;`。
- 这个 heredoc 用单引号 `<<'EOF'` 是**故意的**：块里有 nginx 自己的变量 `$uri`，不能让 shell 展开。所以这里不能改成展开式 heredoc，`SUB_PATH` / 域名用手工替换。
- 如果 VPS 没有 IPv6，删掉两处 `listen [::]:80 ...;`，否则在 IPv6 被禁用的内核上 `nginx -t` 会报 `Address family not supported`。
- 用 Cloudflare Tunnel 走加密订阅时（见「订阅传输安全」），cloudflared 会带上你配置的订阅 Host 回源到本机 80，正好命中这个订阅 server；如果改用一个不在 `server_name` 里的 Host 回源，会被默认 server 404，记得把那个 Host 也加进 `server_name`。

关闭 Nginx 版本暴露：

```bash
if grep -q '^[[:space:]]*server_tokens ' /etc/nginx/nginx.conf; then
  sed -i 's/^[[:space:]]*server_tokens .*/    server_tokens off;/' /etc/nginx/nginx.conf
else
  sed -i '/^[[:space:]]*http[[:space:]]*{/a\    server_tokens off;' /etc/nginx/nginx.conf
fi
```

检查并重载：

```bash
nginx -t
systemctl reload nginx
```

## CF-Vless-WS 保底部署（即 Argo 隧道节点）

这一节就是俗称的「Argo 隧道节点」——用 `cloudflared`（Cloudflare/Argo 隧道）把一条 VLESS-WS 接出去。它是「大保底」用的，不替代前面的四条主力节点。

### sing-box 本地入站

在 VPS 上新增一个只监听本机的 VLESS WebSocket 入站。**它要作为 `inbounds` 数组里的第 5 个元素加进去**，在最后那个 `ss-in` 对象后面补一个逗号再追加（`ss-in` 是四协议里的最后一个入站），不要替换整个文件：

```json
{
  "type": "vless",
  "tag": "cf-vless-ws-in",
  "listen": "127.0.0.1",
  "listen_port": 28080,
  "users": [
    {
      "uuid": "CF_VLESS_UUID"
    }
  ],
  "transport": {
    "type": "ws",
    "path": "CF_WS_PATH"
  }
}
```

加完一定要重新校验，逗号漏了会让整个 config 解析失败：

```bash
sing-box check -c /etc/sing-box/config.json
systemctl restart sing-box
```

推荐变量：

```text
# 下面两个是模板已写死的默认值（127.0.0.1:28080），保留即可，正文没有同名 token 可替换。
# 如果要改，必须同步改 sing-box 的 listen/listen_port 和 cloudflared 的回源地址。
CF_WS_LISTEN = 127.0.0.1
CF_WS_PORT = 28080

# 下面这些才是 CF-Vless 章节需要替换 / 填写的变量。
CF_WS_PATH = /cf-vless-随机字符串
CF_HOSTNAME = cf-vps.example.com
CF_VLESS_UUID = sing-box generate uuid（可与主 VLESS 不同）
CF_SUB_HOSTNAME = cf-sub.example.com   # 仅「订阅传输安全」方式 A 用：订阅专用域名，回源到本机 80。方式 B / 不走 CF 订阅可不填
```

为什么只监听 `127.0.0.1`：

```text
这个入站只给 cloudflared 本机转发使用
不直接暴露公网端口
减少被扫到的概率
```

### cloudflared

安装：

```bash
# 极简镜像可能没有 keyrings 目录，先建好再写入
mkdir -p --mode=0755 /usr/share/keyrings

curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg \
  | tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null

echo "deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared any main" \
  > /etc/apt/sources.list.d/cloudflared.list

apt update
apt install -y cloudflared
cloudflared --version
```

Cloudflare Tunnel 有两种常用绑定方式，**二选一，不要两种都做**。两种方式都会装/管同一个 `cloudflared` 服务，混用会互相覆盖配置、产生冲突。选定一种后，跳过另一种的全部步骤。

```text
方式一 = Dashboard 建 Tunnel + token 安装（cloudflared service install）。配置在 Cloudflare 云端，VPS 上不写 config.yml。
方式二 = 本机 cloudflared tunnel login/create + 自己写 config.yml + 自建 systemd。配置在 VPS 本地。
两者都用 token / 都装一遍，会让 cloudflared 服务定义冲突，二选一即可。
```

方式一：Cloudflare Dashboard 创建 Tunnel，复制 token 后在 VPS 上安装服务。WS 大保底只需要第一条 public hostname：

```text
Hostname: CF_HOSTNAME
Service:  http://127.0.0.1:28080
```

如果还要用「订阅传输安全」的 Cloudflare 加密订阅，在 Dashboard 里**再加一条 public hostname**：

```text
方式 A（推荐，另起订阅专用域名）：
  Hostname: CF_SUB_HOSTNAME
  Service:  http://127.0.0.1:80
  # 并在 Nginx 订阅 server 的 server_name 里加上 CF_SUB_HOSTNAME

方式 B（共用 CF_HOSTNAME，按 path 分流，注意顺序：精确的 WS path 在前）：
  Hostname: CF_HOSTNAME  Path: CF_WS_PATH   Service: http://127.0.0.1:28080
  Hostname: CF_HOSTNAME（不填 Path，兜底）  Service: http://127.0.0.1:80
  # 并在 Nginx 订阅 server 的 server_name 里加上 CF_HOSTNAME
```

```bash
cloudflared service install CLOUDFLARE_TUNNEL_TOKEN
systemctl enable --now cloudflared
systemctl is-active cloudflared
```

方式二：在 VPS 上用 Cloudflare 登录证书创建命名 Tunnel：

```bash
cloudflared tunnel login
cloudflared tunnel create TUNNEL_NAME
# 上一条会打印 "Created tunnel TUNNEL_NAME with id <UUID>"，并把凭证写到
# ~/.cloudflared/<UUID>.json（root 执行就是 /root/.cloudflared/）。
# 记下这个 <UUID>，下面所有 TUNNEL_ID 占位符都替换成它。

# 把凭证挪到 /etc/cloudflared/，否则 systemd（User=root）按 config 里的
# /etc/cloudflared/TUNNEL_ID.json 找不到凭证、服务起不来。
mkdir -p /etc/cloudflared
cp /root/.cloudflared/TUNNEL_ID.json /etc/cloudflared/TUNNEL_ID.json
chmod 600 /etc/cloudflared/TUNNEL_ID.json
```

创建 `/etc/cloudflared/config.yml`。基础版（只做 WS 大保底，不走 Cloudflare 加密订阅）：

```yaml
tunnel: TUNNEL_ID
credentials-file: /etc/cloudflared/TUNNEL_ID.json
protocol: auto
# auto 会优先 QUIC，若机房拦截 cloudflared 出站 UDP/7844，会回退 HTTP/2。
# 如果仍然连不上，可改成 protocol: http2 强制走 HTTP/2/TCP。
loglevel: info
ingress:
  - hostname: CF_HOSTNAME
    service: http://127.0.0.1:28080
  - service: http_status:404
```

如果同时要走「订阅传输安全」的 Cloudflare 加密订阅，二选一替换上面的 `ingress`：

```yaml
# 方式 A（推荐）：订阅走单独域名 CF_SUB_HOSTNAME，回源到本机 Nginx(80)
ingress:
  - hostname: CF_HOSTNAME          # WS 大保底
    service: http://127.0.0.1:28080
  - hostname: CF_SUB_HOSTNAME      # 加密订阅
    service: http://127.0.0.1:80
  - service: http_status:404
```

```yaml
# 方式 B：共用 CF_HOSTNAME，按 path 分流。精确的 WS path 规则必须排在兜底规则前面，
#         否则订阅兜底会把 WS 流量也吞掉。
ingress:
  - hostname: CF_HOSTNAME
    path: CF_WS_PATH               # 只有 WS 路径走 28080
    service: http://127.0.0.1:28080
  - hostname: CF_HOSTNAME          # 其余（含 SUB_PATH）走 Nginx(80)
    service: http://127.0.0.1:80
  - service: http_status:404
```

> 别忘了把回源到 80 用的那个 Host（方式 A 是 `CF_SUB_HOSTNAME`，方式 B 是 `CF_HOSTNAME`）加进 Nginx 订阅 server 的 `server_name`，否则会被默认 server 404。

把 DNS 记录绑定到 Tunnel（方式 A 两个域名都要绑）：

```bash
cloudflared tunnel route dns --overwrite-dns TUNNEL_NAME CF_HOSTNAME
# 方式 A 额外执行：
cloudflared tunnel route dns --overwrite-dns TUNNEL_NAME CF_SUB_HOSTNAME
```

创建 systemd 服务。注意 `cloudflared` 的可执行路径随安装方式不同——`apt install cloudflared` 通常装到 `/usr/bin/cloudflared`，不是 `/usr/local/bin/`。所以下面用 `command -v` 自动定位，不要硬编码路径：

```bash
# 1. 先校验 ingress 配置合法（path/service 写错会在这里直接报出来，不用等服务起不来才发现）
cloudflared tunnel --config /etc/cloudflared/config.yml ingress validate

# 2. 自动定位 cloudflared 实际路径
CFD=$(command -v cloudflared)
echo "cloudflared 路径: $CFD"   # 应为非空，通常是 /usr/bin/cloudflared

# 3. 用定位到的路径生成 service 文件（unquoted EOF，让 ${CFD} 展开）
cat >/etc/systemd/system/cloudflared.service <<EOF
[Unit]
Description=Cloudflare Tunnel
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=root
ExecStart=${CFD} --no-autoupdate --config /etc/cloudflared/config.yml tunnel run
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
EOF
```

启动：

```bash
systemctl daemon-reload
systemctl enable --now cloudflared
systemctl is-active cloudflared
```

> `TUNNEL_ID.json` 凭证只在 `cloudflared tunnel create` 时生成一次，无法重新下载。丢了只能重建 Tunnel。记得纳入备份（见“修改前备份”），且不要外传。

不管用 token 方式还是命名 Tunnel 方式，只有 `CF_HOSTNAME` 已经能公网访问后，才把 `CF-Vless` 加入订阅。

## 流量显示和限流

Clash/Mihomo 显示流量靠 HTTP 响应头，不是靠 YAML 内容：

```http
Subscription-Userinfo: upload=0; download=已用字节; total=总量字节; expire=到期时间戳
```

脚本设计要点：

```text
所有参数从 /etc/sing-box-node.env 读，不在脚本里硬编码
按 COUNT_MODE 统计（默认 tx 只算出站，匹配多数商家；代理 rx≈tx，rx+tx 会翻倍误算）
   —— vnstat 统计的是整块网卡流量，SSH、系统更新等也会算进去；偏保守、更安全
只有头内容变化时才写文件并 reload nginx；流量持续增长时每轮 reload 属正常
   —— 因为 Subscription-Userinfo 是 nginx 配置 include，文件变化后需要 reload 才生效
超额停 sing-box 时在 /var/lib/sing-box-node/ 打一个持久标记；只有带标记才会在额度恢复后自动拉起
   —— 手动 systemctl stop sing-box（无标记）不会被定时任务复活
想永久停用这台节点，不再受配额自动恢复影响：
   systemctl disable --now sing-box && rm -f /var/lib/sing-box-node/quota-stopped
网卡找不到时报错到 stderr（cron 已接到 logger，可在 journal 看到），不会静默
```

创建脚本：

```bash
nano /usr/local/bin/traffic_limit.py
```

模板：

```python
#!/usr/bin/env python3
import json
import subprocess
import datetime
import sys
import os

ENV_PATH = "/etc/sing-box-node.env"
HEADER_PATH = "/etc/nginx/snippets/sub_headers.conf"
STATE_DIR = "/var/lib/sing-box-node"
QUOTA_FLAG = os.path.join(STATE_DIR, "quota-stopped")


def load_env(path=ENV_PATH):
    env = {}
    with open(path) as f:
        for line in f:
            line = line.strip()
            if not line or line.startswith("#") or "=" not in line:
                continue
            k, v = line.split("=", 1)
            env[k.strip()] = v.strip().strip('"').strip("'")
    return env


def parse_expire(s):
    return int(datetime.datetime.strptime(s, "%Y-%m-%d %H:%M:%S %z").timestamp())


def current_month_used(data, interface, mode, now=None):
    """返回当月已用字节；接口不存在返回 None。"""
    now = now or datetime.datetime.now()
    iface = None
    for it in data.get("interfaces", []):
        if it.get("name") == interface:
            iface = it
            break
    if iface is None:
        return None
    rx = tx = 0
    for m in iface.get("traffic", {}).get("month", []):
        d = m.get("date", {})
        if d.get("year") == now.year and d.get("month") == now.month:
            rx = m.get("rx", 0)
            tx = m.get("tx", 0)
            break
    if mode == "rx+tx":
        return rx + tx  # 仅商家真按"进+出"计费才用；代理 rx≈tx，这是真实用量的约 2 倍
    if mode == "max":
        return max(rx, tx)
    return tx  # 默认 tx：只算出站，匹配绝大多数商家


def build_header(used, total, expire):
    return (f'add_header Subscription-Userinfo '
            f'"upload=0; download={used}; total={total}; expire={expire}" always;\n')


def decide_enforcement(used, limit_bytes, active, flag_exists):
    """返回 (action, keep_flag)；action in {None,'stop','start'}。"""
    if used >= limit_bytes:
        if active:
            return ("stop", True)        # 超额且在跑：停掉并打配额标记
        # 已经停了：只有原本就是配额停（有标记）才保留标记；
        # 手动停（无标记）不抢标记，否则下月恢复会被误当配额停机拉起
        return (None, flag_exists)
    if flag_exists:
        return ("start" if not active else None, False)
    return (None, False)  # 无标记的手动停机：不动


def main():
    os.makedirs(STATE_DIR, exist_ok=True)
    env = load_env()
    limit_bytes = int(float(env.get("LIMIT_GB", "200")) * 1024 ** 3)
    interface = env.get("INTERFACE", "eth0")
    mode = env.get("COUNT_MODE", "tx")
    # EXPIRE_AT 缺失/为空时不崩溃，回退 expire=0（多数客户端视为"无到期"）
    expire_raw = env.get("EXPIRE_AT")
    expire = parse_expire(expire_raw) if expire_raw else 0

    try:
        result = subprocess.run(["vnstat", "--json"], capture_output=True, text=True, check=True)
        data = json.loads(result.stdout)
    except Exception as e:
        print(f"Error running vnstat: {e}", file=sys.stderr)
        sys.exit(1)

    used = current_month_used(data, interface, mode)
    if used is None:
        print(f"Interface {interface} not found in vnstat data", file=sys.stderr)
        sys.exit(1)

    header = build_header(used, limit_bytes, expire)
    old = ""
    if os.path.exists(HEADER_PATH):
        with open(HEADER_PATH) as f:
            old = f.read()
    if header != old:
        with open(HEADER_PATH, "w") as f:
            f.write(header)
        subprocess.run(["systemctl", "reload", "nginx"])

    active = subprocess.run(["systemctl", "is-active", "--quiet", "sing-box"]).returncode == 0
    action, keep_flag = decide_enforcement(used, limit_bytes, active, os.path.exists(QUOTA_FLAG))
    if action == "stop":
        subprocess.run(["systemctl", "stop", "sing-box"])
    elif action == "start":
        subprocess.run(["systemctl", "start", "sing-box"])
    if keep_flag:
        open(QUOTA_FLAG, "w").close()
    elif os.path.exists(QUOTA_FLAG):
        os.remove(QUOTA_FLAG)


if __name__ == "__main__":
    main()
```

启用：

```bash

# 前台跑一次：网卡名写错会在这里立刻报 "Interface ... not found"，不会被定时任务隐藏
```

## DNS 和客户端

DNS（先看机器是哪种栈，再决定配哪种记录，详见后面「IPv4 / IPv6 / 双栈」）：

```text
IPv4-only（最常见）：只配 A
  SUB_DOMAIN_1  A     SERVER_IP
  SUB_DOMAIN_2  A     SERVER_IP

IPv6-only：只配 AAAA（前提：客户端网络要有 IPv6）
  SUB_DOMAIN_1  AAAA  SERVER_IPV6
  SUB_DOMAIN_2  AAAA  SERVER_IPV6

双栈：按需配；建议拆两个域名各配一种，避免客户端乱选 IPv6
  node4.example.com  A     SERVER_IP
  node6.example.com  AAAA  SERVER_IPV6
```

如果 DNS 在 Cloudflare 托管，当前直连方案建议先用灰云 `DNS only`。

客户端添加订阅时，名称和 URL 分开填：

```text
订阅名称: AIRPORT_NAME
订阅 URL: http://SUB_DOMAIN_1SUB_PATH
```

示例：

```text
订阅名称: LINE-01
订阅 URL: http://node.example.com/sub-a8f3c91d7e4b4c6f.yaml
```

如果客户端没有单独的订阅名称输入框，就在添加后把配置名称改成 `AIRPORT_NAME`。

不要把订阅名称写成 URL，也不要写成节点协议名。协议名已经用于节点：

```text
Hysteria2
AnyTLS
Vless
SS2022
```

默认不要用 HTTPS 订阅：

```text
443/tcp = VLESS Reality
80/tcp  = Nginx 订阅
```

443 已经留给 VLESS Reality，VPS 本机不再单开 443 做 HTTPS 订阅。如果确实想要加密订阅，按“订阅传输安全”一节，用单独的订阅域名 `CF_SUB_HOSTNAME`（或对 `CF_HOSTNAME` 按 path 分流）经 Cloudflare Tunnel 回源到本机 Nginx(80) 走 HTTPS。注意订阅不能直接打 `CF_HOSTNAME`——那个 Host 回源的是 VLESS WS 入站。

## IPv4 / IPv6 / 双栈

前面的主线模板默认按 **IPv4** 写（节点 `server:` 用 `SERVER_ADDRESS`，IPv4 默认让 `SERVER_ADDRESS=SERVER_IP`；DNS 配 `A`，`ipv6: false`，Nginx `listen 80` + `[::]:80`，分流用 `IP-CIDR,.../32`）。绝大多数 VPS 这样就够了。只有当机器是纯 IPv6 或双栈、且你确实要利用 IPv6 时，才参考下面按情况调整。

为不打乱主线，这里**不重命名** `SERVER_IP`（它仍是 IPv4 字面量，只给 IPv4/双栈分流规则用）；需要 IPv6 时按下面补充变量即可：

```text
SERVER_IP        主线 IPv4 地址（IPv4/双栈机器有，纯 IPv6 机器没有）；
                 IPv4/双栈 rules 的 IP-CIDR/32 用它，纯 IPv6 改用 IP-CIDR6/128
SERVER_IPV6      机器的公网 IPv6，仅 IPv6/双栈用到；rules 的 IP-CIDR6/128 用它
SERVER_ADDRESS   节点连接地址（主模板 server: 已改用它）。IPv4 默认填 SERVER_IP；
                 IPv6-only 填域名 SUB_DOMAIN_1，避免写 IPv6 字面量
```

一句话总览：**IPv4-only 最简单；双栈要防客户端乱选 IPv6；IPv6-only 最好用域名，并准备 Cloudflare Tunnel 保底。**

### IPv4-only（最常见，但先确认内核 IPv6 是否启用）

```text
变量：SERVER_ADDRESS = SERVER_IP（IPv4 字面量）。
DNS：只配 A 记录，不配 AAAA。
Mihomo 模板：ipv6: false 可以保留。
Clash 订阅：server: SERVER_ADDRESS 解析成 IPv4。
分流规则：保留 - IP-CIDR,SERVER_IP/32,DIRECT,no-resolve。

sing-box listen 和 Nginx，看内核 IPv6 栈状态分两种：
  ① IPv6 栈没禁用（默认情况）：listen 用 "::"、Nginx 保留 [::]:80 都没问题，不用改。
  ② IPv6 栈被显式禁用（sysctl disable_ipv6=1）：sing-box listen 必须改 "0.0.0.0"，
     Nginx 必须删掉 listen [::]:80 只留 listen 80，否则服务起不来 / 绑定失败。
```

### IPv6-only

```text
变量：SERVER_ADDRESS = SUB_DOMAIN_1（用域名，别写 IPv6 字面量）。
DNS：只配 AAAA，不配 A。
前提：客户端网络必须有 IPv6，否则手机/电脑根本连不到这台 VPS（这是最大门槛）。
Mihomo 模板：必须改 ipv6: true，否则 AAAA 解析和 IPv6 连接被禁用，照抄会连不上。
sing-box：listen 用 "::"。
Nginx：保留 listen [::]:80；有 IPv4 栈时 listen 80 留着也无妨。
Clash 订阅：server: SERVER_ADDRESS 已是域名 SUB_DOMAIN_1。
分流规则：用 - IP-CIDR6,SERVER_IPV6/128,DIRECT,no-resolve（原来的 /32 只适合 IPv4）。
订阅 URL：用域名 http://node.example.com/sub-xxx.yaml，不要写 http://[IPv6]/...（兼容性差）。
```

### 双栈 IPv4 + IPv6

```text
变量：SERVER_ADDRESS = SERVER_IP（默认走 IPv4）；要走 IPv6 则填对应域名。
Mihomo 模板：想让节点走 IPv6 时改 ipv6: true；只把 IPv6 当备用测试可保留 false。
DNS：可同时配 A 和 AAAA。
风险：客户端可能优先走 IPv6，若其 IPv6 链路不稳，会出现"有时连得上有时连不上"。
稳妥做法：拆两个域名分别只配一种记录，例如
         node4.example.com 只配 A（SERVER_ADDRESS=node4...），
         node6.example.com 只配 AAAA（SERVER_ADDRESS=node6...）。
         共用一个域名时，可参考 mihomo 代理通用字段的 ip-version
         强制选栈，常用值如 ipv4 / ipv6 / ipv4-prefer / ipv6-prefer。
省事做法：主线先只用 IPv4，IPv6 当备用测试。
分流规则：两条都加
         - IP-CIDR,SERVER_IP/32,DIRECT,no-resolve
         - IP-CIDR6,SERVER_IPV6/128,DIRECT,no-resolve
```

> 开启 `ipv6: true` 后补本地 IPv6 直连规则：主模板的私网/本地直连段只有 IPv4（`127.0.0.0/8`、`10/8`、`192.168/16` 等）。一旦走 IPv6，本地 IPv6 流量会因为没有匹配规则而误走代理。在 `rules` 的私网段附近补上：
>
> ```yaml
>   - IP-CIDR6,::1/128,DIRECT,no-resolve        # IPv6 回环
>   - IP-CIDR6,fc00::/7,DIRECT,no-resolve        # 唯一本地地址 ULA
>   - IP-CIDR6,fe80::/10,DIRECT,no-resolve       # 链路本地
>   - IP-CIDR6,ff00::/8,DIRECT,no-resolve        # 组播
> ```

### 纯 IPv6 机器的保底

纯 IPv6 VPS 直连做节点有门槛——很多客户端网络仍是 IPv4-only，连不到 AAAA。这种机器更适合把 Cloudflare Tunnel / CF-Vless-WS 当**重要保底**（不只是大保底）：

```text
客户端连 Cloudflare 域名，Cloudflare 边缘同时支持 IPv4/IPv6。
VPS 上的 cloudflared 主动向 Cloudflare 出站连接（不需要 VPS 有 IPv4 入站）。
这样 IPv4-only 客户端也能通过 Cloudflare 访问 IPv6-only VPS 后面的 WS 节点。
```

但要清楚边界：HY2 / AnyTLS / Reality 这类**直连**协议不能靠 Cloudflare 普通 DNS 代理救活（Cloudflare 7 层代理只认 HTTP(S)/WS）。纯 IPv6 机器要给 IPv4-only 客户端用，就得走 WS/Tunnel 形态，也就是这里的 CF-Vless-WS。

## 之前改动抽象成通用规则

这些不是某一台 VPS 专属，而是以后都建议这样做：

| 项目 | 通用最终做法 | 原因 |
|---|---|---|
| 文档 | 只保留一个 `节点搭建文档.md` | 避免多个版本混乱 |
| 订阅名称 | 使用 `AIRPORT_NAME` | 多条线路时一眼看出是哪条线路 |
| 订阅路径 | 使用随机路径 | `/sub.yaml` 太容易被扫到 |
| 旧路径 | 返回 404 | 不继续暴露旧订阅 |
| 节点名 | `Hysteria2` / `AnyTLS` / `Vless` / `SS2022` | 简洁，不暴露实现细节 |
| 代理组 | `type: select` | 手动切换最方便排查 |
| 运行参数 | 统一进 `/etc/sing-box-node.env` | 单一来源，避免改了一处忘另一处 |
| 流量计费 | 默认 `COUNT_MODE=tx` | 多数商家只按出站计费；代理 rx≈tx，rx+tx 会把用量翻倍误算、提前半量停机 |
| nginx reload | 头变化才 reload | 流量持续增长时每轮 reload 属正常；无统计变化才跳过 |
| 限流停机 | 标记文件区分“限额停”和“手动停” | 手动停机不被定时任务拉起 |
| 限流报错 | stderr 接 `logger` | 网卡写错等故障可见，不静默失效 |
| 防火墙 | ufw + 云安全组都放行，尤其 UDP | 云安全组默认拒入站、常拦 UDP，是连不上的头号原因 |
| 时间同步 | `timedatectl set-ntp true` | Reality/TLS 握手对时钟敏感 |
| Google Play | 不写专用规则 | 通用模板保持干净，特殊问题单独补 |
| 国内分流 | 常用域名 + `GEOSITE,cn` + `GEOIP,CN` | 减少国内 App 误走代理 |
| 流量头 | 放到 `snippets`，只在订阅路径 include | 避免 404 和首页也带流量头 |
| Nginx 首页 | `return 404` | 不暴露默认欢迎页 |
| Nginx 版本 | `server_tokens off` | 少暴露版本 |
| sing-box 权限 | `600 root root` | 配置里有密码和私钥 |
| 订阅传输 | 默认 HTTP，需加密走 CF Tunnel | 443 留给 Reality；明文订阅靠随机路径降低风险 |
| HTTPS 订阅 | 默认不启用 | 443 留给 VLESS Reality |
| Cloudflare | 默认不用 | 直连四协议更简单、好排查 |
| 大保底 | `CF-Vless-WS` | 只在 IP 直连出问题时使用 |
| 备份 | 含 env / cron / cloudflared 凭证 | 否则迁移后限流失效、隧道无法恢复 |

## 可选增强（一键脚本已实现，手搓按需加）

下面这些都不是四协议主线的必需项，而是**可选件**。我的一键脚本（`install.sh`）里已经把它们全做好了，并用真 sing-box 二进制 `sing-box check` 验证过配置合法。手动部署时，按需挑着加即可；不加也不影响主线。

> **脚本来源 / 本文档的边界**：表里凡是写“脚本 `install.sh xxx`”的（看板页 `panel`、网页管理 `admin`、`warp`、`doctor`、`harden`、`backup/restore`），完整实现都在仓库 **`github.com/shiro1888/sing-box-oneclick`**（`install.sh`）。**光靠这份 md 手搓不出这些复杂可选件**——本文档只保证“主线四协议 + 订阅 + 流量显示/限流”能纯手工复现；上面这些自动化增强请去仓库拿脚本一键装。简单的（禁 BT、obfs、sysctl 调优）md 里给了可手搓的要点。

| 增强 | 作用 | 怎么加 |
|---|---|---|
| **服务端路由**：禁 BT/PT + 拦广告 | 在 sing-box 加 `route` 段：`bittorrent` 直接 reject（防被商家收滥用投诉封机），geosite 广告域 reject | 手搓在 `config.json` 加 `route`；或用脚本默认就开 |
| **HY2 obfs（salamander）** | 给 HY2 的 QUIC 包再裹一层混淆，抗“UDP/QUIC 指纹识别” | 服务端 HY2 入站加 `obfs`，客户端同步加 `obfs/obfs-password` |
| **HY2 端口跳跃** | nftables 把一整段 UDP 端口重定向到真实 HY2 端口，抗运营商按端口限速（**云安全组要放整段 UDP**） | 手搓写 nftables NAT 规则 + systemd；或用脚本一键 |
| **HY2 brutal** | 客户端订阅写死上下行带宽，让 HY2 用暴力拥塞控制顶满带宽 | 客户端 HY2 节点加 `up/down` |
| **WARP 解锁分流** | 机房 IP 被 ChatGPT/流媒体拉黑时，把这些站点单独走 Cloudflare WARP 出口，其余直连 | 比较复杂（wgcf 注册 + sing-box 加 wireguard 出站 + route 分流），**建议直接用脚本 `install.sh warp`** |
| **备份 / 迁移** | 把密钥+参数+证书打包，换机一条命令重建、凭证不变 | 见本文「修改前备份」；脚本是 `backup` / `restore` |
| **SSH 加固** | 仅密钥登录 + 禁密码 + fail2ban（**先确认密钥能登录再禁密码**，别锁死自己） | 手动改 `sshd_config`；或脚本 `install.sh harden`（带防锁死护栏） |
| **doctor 自检** | 一次性核对服务/配置/sysctl/防火墙/时间/证书/订阅可达 | 见本文「检查命令」；脚本是 `install.sh doctor` |
| **可视化看板页** | 一个只读 HTML：看订阅/扫码/复制/每节点二维码/流量到期 | 脚本 `install.sh panel`（纯静态，不开管理端口） |
| **网页管理面板** | 网页上改限额/到期/重启/备份。**只监听 `127.0.0.1` + token + 走 SSH 隧道**，不暴露公网 | 脚本 `install.sh admin`（别把端口改成 `0.0.0.0`） |
| **网络调优 sysctl + BBR** | `rmem_max/wmem_max=16MB`（HY2/QUIC 跑满带宽的前提）+ BBR v1；性能优先按 byJoey BBRv3 Max + 菜单 12，保守机器可停在 BBR v1 / XanMod / 标准版 | 见下方「网络加速」一节（给了可直接跑的命令） |

> 这些增强里，**最值得手动也加一下的是网络调优**（16MB UDP 缓冲对 HY2 吞吐影响很大）和**服务端禁 BT**（防封机）。WARP / 端口跳跃 / 网页管理面板这类复杂的，建议直接用一键脚本，别手搓踩坑。

## 网络加速：sysctl 调优 + BBR v3 性能优先方案

上面「可选增强」提到的网络调优，这里给可直接执行的版本。**强烈建议至少做第一步 sysctl**（免重启、零风险，对 HY2 吞吐提升明显）。本文档的性能优先路线是：确认机器是 KVM / 云服务器 / 独立服务器，并且有控制台/VNC/救援模式后，使用 byJoey BBRv3 选择 **Max**，再开启菜单 **12「疯批模式」**；保守生产机器可停在 BBR v1，或改用 XanMod / 标准版。

### 第一步：sysctl 调优（免重启，强烈建议做）

```bash
cat >/etc/sysctl.d/99-singbox.conf <<'EOF'
# UDP/QUIC 大接收缓冲 —— Hysteria2 关键！不设会被限速并刷 quic-go 缓冲告警（官方推荐 16MB）
net.core.rmem_max=16777216
net.core.wmem_max=16777216
# TCP 缓冲，适配跨境高带宽-延迟积（BDP）链路
net.ipv4.tcp_rmem=4096 87380 16777216
net.ipv4.tcp_wmem=4096 65536 16777216
net.core.netdev_max_backlog=10000
net.core.somaxconn=4096
net.ipv4.tcp_mtu_probing=1
net.ipv4.tcp_slow_start_after_idle=0
net.ipv4.tcp_fastopen=3
# BBR v1（纯 sysctl、免重启、零风险）
net.core.default_qdisc=fq
net.ipv4.tcp_congestion_control=bbr
EOF

sysctl --system
# 验证
sysctl net.core.rmem_max                 # 期望 16777216
sysctl net.ipv4.tcp_congestion_control   # 期望 bbr
```

> 容器型机器（OpenVZ/LXC）共享宿主内核，上面部分 sysctl 可能被限制改不动——失败不影响代理，能改多少算多少。KVM 全部生效。

### 第二步（进阶）：BBR v3 —— 性能优先，但先确认救援能力

第一步已经开了 **BBR v1**。按本文档的性能优先策略，满足条件后继续上 **byJoey BBRv3 Max + 菜单 12**；但它需要**换内核**，所以单独列出，先确认风险边界：

```text
为什么仍要加护栏：
1. BBR 是内核 TCP 拥塞控制，帮的是 AnyTLS / Vless 这类 TCP 节点；
   Hysteria2 走 UDP/QUIC、自带拥塞控制，完全不受 BBR 版本影响。
   本套是 HY2 优先，v3/Max/菜单 12 主要是给 TCP 节点和极限测速加上限。
2. 换内核比 sysctl 风险大：要重启，万一新内核起不来要能进救援。
3. 只有 KVM 能换内核；OpenVZ/LXC 共享宿主内核，换不了。

动手前提：这台 VPS 必须有控制台/VNC 或救援模式（内核起不来能进去救），一台一台来，别批量。
```

保守 x86_64 机器可用 **XanMod 内核**（自带 BBRv3，有官方 apt 源 + 签名，比装某个 GitHub 预编译内核更可信、好维护）。如果你按本文档追求极限性能，直接跳到下面 byJoey 段，走 Max + 菜单 12：

> 适用范围：下面这段**只给 x86_64 / KVM 机器**用，装的也是 x86-64 专用包（`linux-xanmod-x64v3`）。
> arm64 机器不要照抄——XanMod 这个源没有对应包；arm64 想要 BBRv3 需另找内核，否则保持发行版内核的 `fq + bbr`（第一步 sysctl 已配）即可，多数场景够用。

```bash
# 0) 看 CPU 指令集等级（决定装哪个包：v1/v2/v3/v4）
wget -qO - https://dl.xanmod.org/check_x86-64_psabi.sh | bash

# 1) 加 XanMod 官方源（gpg --dearmor 需要 gnupg；基础依赖没装，这里先补上。
#    keyrings 目录极简镜像可能也没有，一并建好）
apt install -y gnupg
mkdir -p --mode=0755 /usr/share/keyrings
wget -qO - https://dl.xanmod.org/archive.key | gpg --dearmor -o /usr/share/keyrings/xanmod-archive-keyring.gpg
echo 'deb [signed-by=/usr/share/keyrings/xanmod-archive-keyring.gpg] http://deb.xanmod.org releases main' \
  > /etc/apt/sources.list.d/xanmod-release.list

# 2) 装内核（x64v3 是 CPU 指令集等级、不是 BBR 版本；按第 0 步结果选 v1/v2/v3）
apt update && apt install -y linux-xanmod-x64v3

# 3) 重启进新内核
reboot

# 4) 回来验证
uname -r                                  # 内核名里应带 xanmod
sysctl net.ipv4.tcp_congestion_control    # 期望 bbr（XanMod 里这个 bbr 就是 BBRv3）
```

> 第一步写的 `/etc/sysctl.d/99-singbox.conf` 里已经是 `fq + bbr`，换上 XanMod 内核后**自动用上 v3**，不用再改 sysctl。
>
### byJoey BBRv3 内核（可选；具体内核版本随上游更新，部署时现场查询）

[byJoey/Actions-bbr-v3](https://github.com/byJoey/Actions-bbr-v3) 现在已经不是单纯放 `.deb` 的仓库，而是一个 **BBRv3 管理脚本**：自动识别 `x86_64` / `aarch64`，从 GitHub Releases 选匹配架构的内核，支持安装/更新、指定版本、状态检查、队列算法切换、亚太 TCP 调优、智能带宽优化和卸载。

> 下面的菜单项、`Max` / 标准版命名、`*-max` 资产名等都是**写文档时该脚本的行为**；它是第三方项目、随时可能改。实际**以你运行时 byJoey 仓库的菜单和 Releases 命名为准**，下述名称仅供参考。

适用前提仍然不变：

```text
Debian 12+ / Ubuntu 24.04+
apt-get
x86_64 或 aarch64
KVM / 云服务器 / 独立服务器
建议 GRUB 引导
```

常用菜单：

```text
5  启用 BBR + FQ_CODEL
6  启用 BBR + FQ_PIE
7  启用 BBR + CAKE
8  亚太机器 TCP 调优
10 BBR v3 智能带宽优化
11 清空网络优化配置
12 BBR v3 疯批模式（极限测速挑战）
```

> 本文档的 byJoey 策略是 **默认选 Max 版，并开启菜单 12「疯批模式」**：目标是优先吞吐和极限测速，愿意承担比标准版更激进的拥塞控制和网络调参。菜单 **「1 安装最新版」** 会按提示让你选择标准版或 Max 版；按本文档策略选 **Max**。如果这台 VPS 是保守生产用途、怕链路抖动或怕内核风险，再改选标准版并跳过菜单 12。
>
> 注意：菜单 **12「疯批模式」** 是另一层额外激进的网络调参/测速模式，不等于 Max 内核本身。按本文档的性能优先策略，装好 Max 后可以继续开启 12；如果开启后出现丢包、网页卡顿、SSH 抖动或视频加载忽快忽慢，用菜单 **11「清空网络优化配置」** 回滚这层调参，再保留 Max 内核观察。

选择内核时默认选 **Max 版**，标准版作为稳妥回退：

```text
Max：本文档默认；提高 Startup / ProbeBW / cwnd 进攻性，追求更高吞吐
标准版：更保守；如果 Max 出现抖动、丢包敏感或重启后不稳，再回退标准版
```

版本策略：

```text
文档不要写死某个 7.x.x 版本号。
这里说的“最新内核”，指 GitHub Releases 里当前最新的、匹配本机架构的 Max 版 BBRv3 内核。
让 byJoey 脚本动态读取 GitHub Releases；或自己解析 Releases 时按下面规则过滤：
  x86_64 机器：默认取 x86_64-*-max；
  aarch64/arm64 机器：默认取 arm64-*-max；
  如果要保守稳定：才改取不带 -max 的标准版；
  菜单 12「疯批模式」按本文档性能优先策略开启；它不是内核类型，和 Max 内核分开看。
不要无脑使用 /releases/latest，因为 latest 可能指向另一个架构，或指向不符合当前策略的标准版 / Max 版。
```

也就是说，文档里的目标不是“安装某个固定版本”，而是“安装 byJoey GitHub Releases 当前最新的匹配版本”。具体版本号会随仓库更新而变化，部署时现场查询。

> 对已经装了 XanMod 且 `modinfo tcp_bbr` 显示 `version: 3` 的 x86_64 机器，不需要只因为 byJoey 有更新就焦虑换内核。若按本文档的性能优先路线，确认控制台/VNC/救援模式可用后，再切 byJoey Max + 菜单 12；若这台机器更看重长期稳定，就保留 XanMod 或改用 byJoey 标准版。

## 什么时候再加东西

暂时不建议把 Cloudflare WS / Argo / WARP 当主力。它们适合作为特殊备用，不适合作为这个模板的主线。

暂时不建议默认加 TUIC v5。除非 HY2 在某些网络明显不稳定，再作为额外的 UDP 直连备用（主线已有四条协议）。

暂时不建议默认加自动测速组。手动 `select` 更适合自用排查。
